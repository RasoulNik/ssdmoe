What has been tried already
==================================================

Keep these outcomes in mind so you do not waste time repeating them without a new reason.

Validated wins:

- native expert `pread()` loop beats Python read loop
- concurrent per-component native reads materially help
- `mx.asarray(np_buffer)` is slightly better than `mx.array(np_buffer)` in the hot conversion path
- lower `K` is the strongest validated throughput lever

Validated non-wins / dead ends:

- `F_NOCACHE` hurt end-to-end generation
- larger explicit expert caches triggered memory pressure on this 16 GB machine
- resident expert scales/biases are mixed at best and not the default path
- speculative decoding with local Qwen3.5-4B draft model was slower, not faster
  - artifact: /Users/rasoul/Documents/projects/MLX/.run/specdec-k2-draft4b.json
- a single native multi-component read call regressed versus the existing concurrent per-component path
- a pooled native “materialized views” implementation also regressed and was reverted
  - artifacts:
    - /Users/rasoul/Documents/projects/MLX/.run/native-views-k8.json
    - /Users/rasoul/Documents/projects/MLX/.run/native-views-k2.json

Compression plausibility check:

- Apple `compression_tool` on real quantized expert slabs showed that chunked compression is not obviously impossible
- but CPU-side decompression throughput is not yet clearly fast enough to be a guaranteed win
- compression is secondary until the execution/materialization path is tighter

==================================================
Current bottleneck model
==================================================

This is the current best interpretation of the measurements:

1. The machine is not blocked by raw arithmetic peak alone.
2. The pipeline is not blocked by raw SSD headline bandwidth alone.
3. The real limit is the full path:

   SSD -> native read -> materialization / conversion -> routed expert qmm execution

4. At higher `K`, read time is still dominant.
5. At lower `K`, read time and expert compute become comparable.
6. That means the next high-upside work is not more Python tuning. It is:

   - native expert materialization or reusable quantized buffers that actually win end-to-end
   - a fused routed-expert execution boundary via MLX extension or Metal
